# Golden Dataset — Provenance

`golden_dataset_v1.json` contains 80 hand-verified test cases, 20 per category
(`billing`, `account`, `general`, `technical`). Every case is sourced from
real data — nothing in this file was generated by an LLM.

## Where the data comes from

| Category | Source | How | Count |
|---|---|---|---|
| billing, account, general | [Bitext customer support dataset](https://github.com/bitext/customer-support-llm-chatbot-training-dataset) (CDLA-Sharing-1.0) | Real intents remapped to our 4-category schema; entity placeholders (`{{Order Number}}`, etc.) filled with realistic sample values | 60 |
| technical | Public GitHub issue trackers (Signal Desktop, GitHub Desktop) | Real, live bug reports fetched via the GitHub REST API, cleaned of issue-template scaffolding | 20 |

## How ground truth was assigned

- **Category labels** (billing/account/general): derived from Bitext's own
  human-assigned `intent` field via a fixed mapping in `intent_map.py` — not
  re-labeled by a model.
- **Category label** (technical): all 20 are genuine bug reports from
  consumer software issue trackers; category is unambiguous by construction.
- **Summaries**: for the Bitext-derived cases, one summary template was
  hand-written per real intent (27 total — see `intent_map.py`), then applied
  to every real example of that intent. For the GitHub-derived cases, each
  summary was hand-written after reading the actual issue (see `SELECTED` in
  `build_github_cases.py`). No summary was generated by an LLM.
- **Difficulty tags**: for Bitext cases, derived from the dataset's own
  linguistic flags (`Z` = real typo, `Q` = real colloquial phrasing, `N` =
  negation — see `difficulty_for_flags()` in `build_bitext_cases.py`). For
  GitHub cases, assigned by hand based on how much the report deviates from a
  simple single-symptom bug.

## Known limitation

The Bitext dataset is e-commerce/telecom support, not SaaS bug reports — it
has no native "technical" intents. That's the whole reason the technical
category is sourced separately from GitHub instead of forced into the
Bitext taxonomy.

## How to reproduce this dataset from scratch

```bash
cd dataset_build
export GITHUB_TOKEN=github_pat_...   # avoids the 60/hr unauthenticated rate limit
./fetch_raw_data.sh                  # clones Bitext, fetches GitHub issues into raw/ (gitignored)
python3 build_bitext_cases.py        # -> golden_dataset/bitext_cases.json
python3 clean_github_issues.py       # -> golden_dataset/github_candidates.json
python3 build_github_cases.py        # -> golden_dataset/github_cases.json
python3 merge_golden_dataset.py      # -> golden_dataset/golden_dataset_v1.json
```

`raw/` and the intermediate per-source JSON files are kept out of/optional in
git (the 19MB Bitext CSV especially) since they're fully reproducible from
the commands above. `golden_dataset_v1.json` — the file the eval pipeline
actually consumes — is committed.

## Adding v2

Never edit `golden_dataset_v1.json` in place. To expand the eval bar, add new
cases (e.g. more edge cases discovered from real failures) and write out
`golden_dataset_v2.json`, so eval history stays comparable across dataset
versions the same way it stays comparable across prompt versions.
